The Architecture

The key to speed was a parallel orchestration pattern: one primary AI agent (the "orchestrator") coordinated dozens of specialized background agents, each working on independent features simultaneously. The orchestrator never waited idle — while agents built features in the background, it built features directly.

Two Roles, One Conversation

Orchestrator (Direct Builder)

The primary agent in the conversation. It read files, created controllers and views directly, added routes, wired up quick search and command palette entries, added nav links, verified HTTP 200 responses, and coordinated all work. It never sat idle — while agents ran in background, it built features itself.

Background Agents

Specialized sub-agents launched with detailed prompts. Each received full context about the codebase conventions (API patterns, CSS variables, Stimulus controllers, defensive coding) and was given a complete feature specification. They worked autonomously and reported back when done.

Agent Deployment Strategy

Key Insight: Agents were launched in batches of 2-3 at a time, always for features that touched different files. This avoided merge conflicts. The orchestrator handled shared files (routes.rb, quick_search_controller.rb, command_palette_controller.js, application.html.erb) directly, since those needed sequential edits.

Why It Worked

1. File-Level Isolation

Each agent created its own controller and view files. Shared files (routes, nav, search) were only edited by the orchestrator, eliminating conflicts.

2. Detailed Agent Prompts

Every agent received the full codebase conventions: API client methods, response formats, CSS variables, icon library, chart patterns, error handling. They produced code that matched the existing style.

3. Never Idle

The orchestrator built features directly while agents worked in background. This doubled or tripled effective throughput compared to sequential work.

4. Verify Everything

Every feature was HTTP 200-verified with curl. Agent output was reviewed for correctness. Broken API method calls and missing files were caught and fixed immediately.

5. Defensive Conventions

All agents were instructed to use do...end blocks for Thread.new (not {}), check is_a?(Hash) on API responses, and handle empty data gracefully. This prevented runtime errors.

6. Standing Directive

The user's instruction was simple: "Keep going until I tell you to stop." This eliminated approval overhead and allowed fully autonomous execution across sessions.
